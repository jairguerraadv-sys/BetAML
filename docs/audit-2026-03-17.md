## 4. ANÁLISE DE SEGURANÇA

### 4.1 Riscos CRÍTICOS (Blockers Produção)

#### 🔴 RISCO 1: Secrets em .env Versionado

**Descrição:** Arquivo `.env` com credentials default commitadas no git (JWT_SECRET, PII_ENCRYPTION_KEY, POSTGRES_PASSWORD, REDIS_PASSWORD, MINIO_ROOT_PASSWORD).

**Evidência:**
```bash
$ cat .env | grep SECRET
JWT_SECRET=your-secret-key-min-32-chars-change-this-in-production
PII_ENCRYPTION_KEY=your-32-byte-base64-key-for-aes-256-change-this-in-production
INTERNAL_WEBHOOK_SECRET=default-webhook-secret-change-in-production
```

**Impacto:**
- 🔴 **CRÍTICO:** Se repositório vazar, atacante pode:
  1. Descriptografar todos os CPFs (PII_ENCRYPTION_KEY)
  2. Forjar tokens JWT válidos (JWT_SECRET)
  3. Acessar banco de dados (POSTGRES_PASSWORD)
  4. Acessar buckets S3 (MinIO credentials)

**Remediação:**
1. ✅ Remover `.env` do git: `git rm --cached .env && git filter-repo --path .env --invert-paths`
2. ✅ Adicionar `.env` ao `.gitignore` (já presente)
3. ✅ Migrar secrets para AWS Secrets Manager / Azure Key Vault / HashiCorp Vault
4. ✅ Atualizar `services/api/config.py` para boto3 client fetch
5. ✅ CI/CD injetar secrets via environment variables (GitHub Secrets)
6. ✅ Pre-commit hook com gitleaks/truffleHog para detectar secrets

**Prazo:** 2 dias.

---

#### 🔴 RISCO 2: Refresh Token Rotation Ausente

**Descrição:** Access token JWT de 60min, sem refresh token sliding window. Logout revoga token no Redis, mas polling delay permite reuso.

**Evidência:**
```python
# services/api/config.py:28
access_token_expire_min: int = Field(60, description="JWT expiration minutes")

# services/api/routers/auth.py:89-105
token = create_access_token(payload)  # 60min fixo
# ❌ Sem refresh_token
```

**Impacto:**
- 🔴 **CRÍTICO:** Token comprometido válido por até 60 minutos.
- 🟡 **MÉDIO:** Logout não revoga imediatamente (1-5s de delay no Redis blacklist polling).
- 🟡 **MÉDIO:** Compliance PCI-DSS 8.2.4 requer sessão timeout ≤15min para sistemas PII.

**Remediação:**
1. ✅ Criar refresh_token cookie (httpOnly, 7 dias sliding window)
2. ✅ Schema migration: adicionar `users.refresh_token_jti` column
3. ✅ POST /auth/refresh valida refresh_token, invalida anterior, retorna novo access_token (15min)
4. ✅ POST /auth/logout revoga ambos tokens (access + refresh) no Redis
5. ✅ Middleware valida access_token blacklist antes de processar request

**Prazo:** 1 dia.

---

#### 🔴 RISCO 3: HTTPS/TLS Ausente em Dev

**Descrição:** Docker-compose usa HTTP plain text. Produção requer TLS end-to-end.

**Evidência:**
```yaml
# infra/docker-compose.yml
api:
  ports:
    - "8000:8000"  # ❌ HTTP não criptografado
```

**Impacto:**
- 🔴 **CRÍTICO:** MitM attacks podem capturar JWT tokens, CPFs, passwords.
- 🔴 **CRÍTICO:** Compliance LGPD Art. 46 requer "medidas técnicas adequadas" (TLS obrigatório).
- 🔴 **CRÍTICO:** PCI-DSS 4.1 requer TLS 1.2+ para transmissão de PII.

**Remediação (já implementado no Helm chart):**
1. ✅ Ingress Nginx com cert-manager (Let's Encrypt) — `helm/betaml/templates/ingress.yaml`
2. ✅ PostgreSQL SSL mode=require — atualizar DATABASE_URL
3. ✅ Redis TLS — atualizar REDIS_URL com rediss://
4. ✅ Kafka SASL_SSL — Redpanda config com TLS listener
5. ✅ MinIO TLS — configurar certificados

**Prazo:** 1 dia (config apenas, infra já pronta no Helm).

---

#### 🔴 RISCO 4: PII Logging Não Auditado

**Descrição:** Não verificado se CPF plaintext ou masked está sendo logado em logger.info/error/debug.

**Evidência:**
```bash
$ grep -r "logger\." services/api/ | grep -i cpf
# (não executado ainda, potencial vazamento)
```

**Impacto:**
- 🟡 **MÉDIO:** Logs podem expor CPF plaintext, violando LGPD Art. 46.
- 🟡 **MÉDIO:** CloudWatch/Datadog logs persistem por 90d+, aumentando janela de exposição.

**Remediação:**
1. ✅ Grep codebase: `grep -rn "logger.*cpf" services/`
2. ✅ Validar que apenas `mask_cpf()` é logado (***.***.***.09)
3. ✅ Adicionar ruff custom rule: `# ruff: noqa: PII100` para bloquear `logger.*{cpf}`
4. ✅ Pre-commit hook com regex pattern para detectar logging de fields encrypted

**Prazo:** 0.5 dias.

---

#### 🔴 RISCO 5: Rate Limiting Global (Sem Diferenciação por Role)

**Descrição:** SlowAPI limiter aplica 10 requests/min global em /auth/login, sem distinção de role.

**Evidência:**
```python
# services/api/routers/auth.py:71
@limiter.limit("10/minute")  # ❌ Global para todos roles
async def login(...)
```

**Impacto:**
- 🟡 **MÉDIO:** ADMIN pode ser bloqueado em operações legítimas bulk (ex: onboard 50 users).
- 🟡 **MÉDIO:** Atacante abusa de trial-and-error em AUDITOR accounts (menor privilégio).

**Remediação:**
1. ✅ Configurar `key_func=lambda: f"{request.state.user['role']}:{request.client.host}"`
2. ✅ Limites sugeridos:
   - ADMIN/SUPER_ADMIN: 100/min
   - AML_ANALYST: 50/min
   - AUDITOR: 20/min
   - Anônimo (pre-auth): 10/min (atual)
3. ✅ Documentar em `docs/ops-guide.md` Section 9

**Prazo:** 0.5 dias.
